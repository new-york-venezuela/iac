# AGENTS.md — AI Agent Context for Mailcow IaC

This file gives AI coding agents the context needed to safely assist with this repository.

---

## What This Repo Does

GitOps-driven IaC for deploying:
- **Caddy** (edge proxy, automatic TLS) on a Hetzner CX22 server
- **Mailcow** (Docker Compose email suite) behind Caddy on localhost
- **Static corporate website** served directly by Caddy
- All persistent data on a dedicated 50 GB Hetzner Block Volume

Architecture summary:
```
Internet → Caddy (:80/:443) → static files OR → localhost:8080 (Mailcow)
Mail clients → Mailcow directly (:25/:465/:587/:993)
All Docker data → /mnt/mailcow-data (Hetzner Volume, ext4)
```

## Key Design Decisions

### Why Caddy, not Nginx?
Caddy handles ACME TLS automatically with zero cron maintenance. Mailcow's internal Nginx is left running but bound to `127.0.0.1:8080` only. Port 80/443 are owned by Caddy.

### Why Docker data-root on the volume?
Mailcow stores mail (vmail) and databases as Docker named volumes under `/var/lib/docker/volumes/`. Moving `data-root` to the Hetzner volume ensures **all** persistent data survives server replacement. The volume is detachable and re-attachable independently of the server.

### Why `SKIP_CLAMD=y`?
ClamAV requires ~1-2 GB RAM. On a 4 GB server with swap, this risks OOM kills. Disable for production small instances; enable if upgrading to CX32 (8 GB).

### Why `SKIP_LETS_ENCRYPT=y` in mailcow?
Caddy owns TLS. Mailcow's acme-mailcow module would fight for port 80 ownership. Caddy terminates HTTPS and proxies plain HTTP to mailcow:8080.

## What Agents MUST NOT Do

- **No `terraform apply`** — all cloud actions require explicit human approval
- **No `terraform destroy`** — destructive, data loss
- **No SSH key generation** committed to the repo
- **No hardcoded tokens, passwords, or secrets** in any tracked file
- **No modifications to `.env`** — this file is gitignored for a reason
- **No pushing to main directly** — use PRs

---

## Safe Operations for Agents

- Edit Terraform variables (`variables.tf`) — new params, validation rules
- Edit Ansible playbook tasks — add tasks, fix bugs, update versions
- Edit `Caddyfile.j2` template — add routes, security headers
- Edit documentation (`README.md`, `INSTRUCTIONS.md`, this file)
- Run `terraform validate` and `terraform fmt` (read-only / formatting)
- Run `ansible-lint ansible/playbook.yml` (linting only)

## Common Debugging

```bash
# Check Caddy status and cert issuance
ssh root@<ip> systemctl status caddy
ssh root@<ip> journalctl -u caddy -f

# Check mailcow containers
ssh root@<ip> docker ps -a
ssh root@<ip> cd /opt/mailcow-dockerized && docker compose logs -f

# Verify volume is mounted
ssh root@<ip> df -h /mnt/mailcow-data
ssh root@<ip> ls /mnt/mailcow-data/docker/volumes/

# Test mail ports
nc -zv <ip> 25 465 587 993

# Ansible dry-run
ansible-playbook -i ansible/inventory.ini ansible/playbook.yml --check -e domain=...
```
